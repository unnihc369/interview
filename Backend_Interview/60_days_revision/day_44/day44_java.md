# 1. Why Profiling Matters

```text
Answer flow: Metrics → Profiler → Thread dump → Heap dump → Code fix
```

Tools: **JFR** (Java Flight Recorder), **JMC** (Java Mission Control), **VisualVM**.

# 2. Java Flight Recorder (JFR)

## What It Is

Low-overhead event recorder built into JVM (Oracle/OpenJDK). Captures CPU, memory, GC, I/O, lock contention.

```text
Overhead: ~1% when running — safe for production
```

---

# Starting JFR

```bash
# Start recording (60 seconds)
jcmd <pid> JFR.start name=myrec duration=60s filename=recording.jfr

# Dump while running
jcmd <pid> JFR.dump name=myrec filename=snapshot.jfr

# Stop
jcmd <pid> JFR.stop name=myrec
```

```bash
# At JVM startup
java -XX:StartFlightRecording=duration=60s,filename=app.jfr -jar app.jar

# Continuous (Java 11+)
java -XX:StartFlightRecording=maxsize=100m,maxage=1h,filename=continuous.jfr -jar app.jar
```

---

# Key JFR Events

| Event | Shows |
|-------|-------|
| `jdk.ExecutionSample` | Hot methods (CPU) |
| `jdk.GCPhasePause` | GC pause times |
| `jdk.JavaMonitorEnter` | Lock contention |
| `jdk.SocketRead` / `Write` | I/O bottlenecks |
| `jdk.ObjectAllocationInNewTLAB` | Allocation hotspots |

# 3. Java Mission Control (JMC)

## What It Is

GUI to analyze `.jfr` files. Download with JDK or standalone.

---

# JMC Workflow

1. Open `recording.jfr` in JMC
2. **Automated Analysis** — flags issues (long GC, lock storms)
3. **Method Profiling** — sorted by CPU sample time
4. **Memory** — allocation rates, heap pressure
5. **Threads** — blocked/waiting threads, lock chains

---

# What to Look For

```text
CPU hot spot     → optimize algorithm or cache
High alloc rate  → object churn, consider pooling
Long GC pauses   → heap sizing, leak, or too many short-lived objects
Lock contention  → narrow synchronized blocks, concurrent structures
```

# 4. VisualVM

## What It Is

Free profiler bundled with Oracle JDK (separate install for newer JDKs). Attach to running JVM.

---

# VisualVM Tabs

| Tab | Purpose |
|-----|---------|
| Monitor | CPU, heap, GC activity live |
| Threads | Thread states, dump threads |
| Sampler | CPU/memory sampling (low overhead) |
| Profiler | Instrumentation profiling (higher overhead) |
| Heap Dump | Analyze object retention |

---

# Attach to Process

```bash
# List Java processes
jps -l

# VisualVM: File → Add JMX Connection
# or double-click local application
```

---

# CPU Profiling Example Flow

1. Sampler → CPU → Start
2. Reproduce slow request
3. Stop → see top methods by sample count
4. Drill into call tree

---

# Heap Dump Analysis

```text
Symptom: OutOfMemoryError, growing heap
Steps:
  1. jmap -dump:live,format=b,file=heap.hprof <pid>
  2. Open in VisualVM or Eclipse MAT
  3. Find "biggest objects" / dominator tree
  4. Trace GC roots → leaking collection?
```

# 5. JFR vs VisualVM vs async-profiler

| Tool | Overhead | Production-safe | Best for |
|------|----------|-----------------|----------|
| JFR + JMC | ~1% | Yes | Production, holistic JVM events |
| VisualVM Sampler | Low–Med | Staging mostly | Quick local attach |
| VisualVM Profiler | High | No | Deep local investigation |
| async-profiler | Very low | Yes (attach) | Flame graphs, CPU/allocation |

# 6. Common Production Scenarios

## Slow API under load

```text
1. Check APM (Datadog/New Relic) p99 latency
2. Capture JFR during spike
3. JMC → Method Profiling → top frames
4. Thread dump if blocked: jstack <pid>
```

## Memory leak

```text
1. Monitor heap growth over hours
2. Two heap dumps spaced apart → compare retained sets
3. Fix: unbounded cache, static collection, listener not removed
```

## GC thrashing

```text
JFR → GC events → if Young GC every few seconds with tiny pauses but high CPU
→ increase heap or reduce allocation rate
```

# Interview Questions

## Q1. JFR vs heap dump?

JFR = time-series events (what happened over time). Heap dump = point-in-time object graph (what's in memory now).

## Q2. Can you run profiler in production?

JFR yes (designed for it). Full instrumentation profiler no — use sampling.

## Q3. How to find deadlock in production?

`jstack` or JFR lock events or JMC thread analysis — look for circular lock dependency.

## Q4. What is flame graph?

Stack trace visualization — width = sample count. Hot paths appear wide. Generated by async-profiler or JMC.

## Q5. Safepoint bias?

Sampling profilers only sample at safepoints — can underrepresent some code. JFR + async-profiler mitigate this.
